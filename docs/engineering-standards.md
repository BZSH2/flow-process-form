# 企业工程规范

## 目标

沉淀可复用的工程实践，保证多人协作下的可维护性、可追踪性与可发布性。

## 核心规则

- 变更前先同步主分支，减少冲突。
- 提交信息遵循仓库现有 commit 规范。
- 每个提交聚焦单一主题，避免混杂无关改动。
- 重点关注安全、边界条件、异常处理与可读性。
- 评审意见尽量具体到文件和行为，不做笼统结论。
- 评审通过前不得跳过基础质量检查。
- 所有密钥必须放环境变量，不允许硬编码进仓库。
- 示例配置使用占位值，真实值仅保存在安全环境。
- 对外共享日志时应脱敏处理账号、密钥、令牌信息。
- 发布前确保关键命令可在本地或 CI 复现。
- 部署脚本需可重复执行，避免一次性人工步骤。
- 生产变更应具备可回滚路径，优先采用镜像版本回退。
- 新增工程流程时同步更新根目录 `README.md` 的文档导航。
- 变更 CI/CD 或质量门禁时优先更新对应专题文档。
- 文档中的命令应保持可直接复制执行。

## 落地检查

- 合并前执行 `pnpm lint`、`pnpm typecheck`、`pnpm build`
- 涉及发布链路改动时同步更新 CI/CD 文档
